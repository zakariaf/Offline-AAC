## Why this exists

Users at 200%+ text scale are the target audience, not an edge case. A tile whose label is clipped mid-shutdown is this product failing the exact person it exists for — and nobody will report it, because there is no telemetry and a user who cannot speak does not file bugs. Worse, the clipping class that matters here is **silent**: `RenderParagraph` has no overflow indicator, so a label overrunning its tile produces zero errors and a green suite. This matrix and its fit assertion are the only thing standing between that user and an unreadable board.

## Scope

Build the scale matrix on top of the harness from E05-T07 (`test/support/harness.dart`: `Device`, `tester.useDevice`, `tester.pumpApp({textScaler, boldText, …})`).

**1. `test/ui/overflow_matrix_test.dart` — the loud class.**

The full tuple space: `Device.all` × `[1.0, 1.3, 1.5, 2.0, 3.0]` × `[false, true]` bold = **30 tests** for the board. One `testWidgets` per tuple, generated by a `for` loop **around** the `testWidgets` call — never a loop inside one test.

```dart
const List<double> scales = <double>[1.0, 1.3, 1.5, 2.0, 3.0];

for (final Device device in Device.all) {
  for (final double scale in scales) {
    for (final bool bold in <bool>[false, true]) {
      testWidgets('board: no overflow @ $device x$scale${bold ? ' bold' : ''}',
          (WidgetTester tester) async {
        tester.useDevice(device);
        await tester.pumpApp(
          textScaler: TextScaler.linear(scale),
          boldText: bold,
        );
        expect(tester.takeException(), isNull,
            reason: 'tile text overflowed at $device x$scale');
      });
    }
  }
}
```

Devices come from the harness and are pinned in **logical** size × DPR (`view.physicalSize` is physical pixels): `Device.seLike` 320×568 @ 2.0, `Device.small` 360×800 @ 3.0, `Device.pixel7` 412×915 @ 2.625. `useDevice` already does `addTearDown(view.reset)`.

`TextScaler.linear` is a deliberate over-approximation — Android 14+ scales large text *less* than small text, so linear stresses 20pt tile labels harder than a real device would. Wanted conservatism; do not describe these tests as device-faithful in comments or commit messages. 1.3 and 1.5 are in the list because nonlinear device scaling makes the mid-range the non-obvious part. 3.0 is Larger Accessibility Sizes territory.

`boldText` is a first-class axis paired with every scale, never tested alone: `wght` changes advance widths, so a tile that fits at 2.0x unbolded can overflow at 2.0x bold. It is a system accessibility setting this audience turns on. Note that per `reed-typography`, weight is frozen across all four palettes and `boldText` is the *only* thing permitted to move it — so this axis is the only weight variation the layout ever sees.

**2. Run the same tuple loop separately over show-text mode, edit mode, and the type-to-speak field.** Overflow only reports if the widget **paints** — `Offstage` subtrees and off-viewport content never report, so a board test gives these zero coverage. Show mode fits its text by construction; the honest degradation past six lines at the 32pt floor is a scrollable block, so assert **that**, not the absence of scroll. The type-to-speak field is the 13th cell and grows with the same scaler — give it its own tuple loop.

**3. `test/ui/label_fit_test.dart` — the real gate.**

There is **no minimum tap-target dp floor in this app**, and reintroducing one is a regression. WCAG 2.5.8 (AA) is 24×24 CSS px, 2.5.5 Enhanced (AAA) is 44×44; any larger invented number (76dp is the usual one) appears in no standard, is non-binding — tiles compute to ~89–106 × 125–146dp — and at 200% it measures the wrong thing, because the label block needs ~124dp against ~125dp of tile. The tile is exactly `(viewport − chrome) / rows`. The only question is whether the label fits inside it.

```dart
int linesOf(WidgetTester tester, String label) => tester
    .renderObject<RenderParagraph>(find.text(label))
    .computeLineMetrics()
    .length;

for (final double scale in <double>[1.0, 1.3, 1.5, 2.0, 3.0]) {
  testWidgets('every starter label fits its tile @ x$scale',
      (WidgetTester tester) async {
    tester.useDevice(Device.small); // 360x800: the tightest shipped 3x4
    await tester.pumpApp(textScaler: TextScaler.linear(scale));

    expect(tester.takeException(), isNull); // the loud class

    for (final TestTile t in kTestTiles) {
      final Rect slot =
          tester.getRect(find.byKey(ValueKey<String>('slot_${t.row}_${t.col}')));
      final Size text = tester.getSize(find.text(t.label));

      // Tile inset is 16dp on every side. The label is bottom-anchored and
      // start-aligned; overrun leaves NO exception, only clipped words.
      expect(text.height, lessThanOrEqualTo(slot.height - 32.0),
          reason: '"${t.label}" needs ${text.height}dp inside a '
              '${slot.height}dp tile at x$scale — it is being clipped silently');
      expect(text.width, lessThanOrEqualTo(slot.width - 32.0),
          reason: '"${t.label}" is ${text.width}dp wide at x$scale');

      // A literal \n in a shipped label is a HINT, not a guarantee: past 3 lines
      // the layout falls back to natural wrap. Assert the ceiling holds.
      expect(linesOf(tester, t.label), lessThanOrEqualTo(3),
          reason: '"${t.label}" wrapped to >3 lines at x$scale');
    }
  });
}
```

Run this loop with `boldText: true` as well.

**4. `test/ui/text_scale_honored_test.dart` — the anti-clamp behavioural check.** The grep in step 5 catches the named API; this catches a clamp built by hand.

```dart
testWidgets('text scale is honored, never clamped', (WidgetTester tester) async {
  tester.useDevice(Device.small);

  await tester.pumpApp();
  final double base = tester.getSize(find.text('Overwhelmed')).height;

  await tester.pumpApp(textScaler: const TextScaler.linear(2.0));
  final double scaled = tester.getSize(find.text('Overwhelmed')).height;

  // 1.8, not 2.0: tolerate line-height rounding, still fail hard on a clamp.
  expect(scaled, greaterThan(base * 1.8),
      reason: 'Text did not grow at 2.0x. Someone clamped TextScaler to keep the '
          'fixed grid tidy. 200%+ must be honored.');
});
```

Contrast and tap-target both stay green while text stops growing — no guideline catches this.

**5. `test/policy/no_text_clamping_test.dart` — make the ban structural.** Discipline is exactly what the no-telemetry constraint says cannot be relied on, and no lint covers any of this. Walk `Directory('lib')` recursively and fail on `withClampedTextScaling`, `textScaleFactor`, `FittedBox`. Walk `Directory('test')` and fail on `ignoreOverflowErrors`, `FlutterError.onError =`. A single hit disarms the whole matrix.

**Also run the matrix against the 6-tile layout.** Layout is a user setting chosen once. At 200% a 20pt label becomes 40pt and 12 tiles arithmetically cannot render on a phone; the 6-tile layout is a legitimate resolution, and it must never be the escape hatch nobody tests. Never run against 3×4 alone.

**Out of scope:**
- **Goldens.** None for v1. 4 palettes × 5 scales × 2 modes = 40 PNGs, each invalidated by any padding tweak, and a golden cannot assert anything — one rendering with clipped text passes forever once blessed.
- Any minimum tap-target dp assertion. See above.
- Rebuild counts, `RepaintBoundary`, frame timing — zero animation means there is no frame budget to miss.
- Fixing a red matrix by changing `lib/` layout code (that is a separate task); this task lands the tests and reports what is red.

## Traps

- **Swallowing the net you already have.** `RenderFlex` overflow already **fails** a widget test — `DebugOverflowIndicatorMixin._reportOverflow` routes through `FlutterError.reportError`, the binding captures it into `_pendingExceptionDetails`, and `testWidgets` rethrows at test end. The entire blog genre on this topic is about how to *suppress* it. The job is not to make overflow fail; it is to never lose the net. Never `takeException()` to swallow, never assign `FlutterError.onError` in a layout test, never copy the popular `ignoreOverflowErrors` helper.
- **`takeException()` in a global `tearDown`.** It clears `_pendingExceptionDetails` before `testWidgets` rethrows, silently converting the whole suite's overflow net into a no-op. Every matrix test then passes while checking nothing.
- **Believing `takeException(), isNull` is sufficient.** It is necessary and *not* sufficient. A clipped `Text` reports nothing, ever — only a `Flex` child reports. A label running past a fixed-height `SizedBox` + clip gives zero errors, a green test, and unreadable words on a real phone. The fit assertion is the actual gate.
- **Looping scales inside one `testWidgets`.** Overflow is reported **once per `RenderObject`** — `_overflowReportNeeded` goes false after the first report and resets only on `reassemble()`. A loop inside a test silently under-reports scales 2..n. One `testWidgets` per tuple, always.
- **The 800×600 default surface.** Wider than any phone. Unpinned, tiles come out ~2× too wide, the text fits, the suite is green, and the shipped 360dp phone is broken.
- **Setting `physicalSize` without the DPR.** `view.physicalSize = Size(320, 568)` at the default DPR of 3.0 gives a 107×189 surface, not a small phone. `useDevice` multiplies by DPR — do not hand-roll it.
- **Leaking the view size.** Skip `addTearDown(view.reset)` and the next test in the file inherits your surface; the failure lands in a file nobody edited. Prefer `TestFlutterView.reset()` over the `resetPhysicalSize` / `resetDevicePixelRatio` pair.
- **Assuming the board test covers show mode and edit mode.** Overflow only reports if the widget **paints**. They do not paint under a board test, so they have no overflow coverage at all unless pumped separately. (Clipped content *does* still report — `RenderFlex` calls `paintOverflowIndicator` after pushing its own clip.)
- **The four wrong fixes for a red matrix.** Each turns the test green and the product worse:
  - `MediaQuery.withClampedTextScaling` / `textScaleFactor` — the one-line "fix" a tired contributor applies at 2am. It defeats the entire matrix while contrast and tap-target still pass green, and it overrides the user's own OS setting — the setting they need.
  - `FittedBox` / any auto-shrink — backwards: it makes the **longest** (most complex) phrase the **smallest**, destroys the grid's uniform rhythm, and silently cancels the user's `TextScaler`.
  - `TextOverflow.ellipsis` / `maxLines` truncation — an ellipsis on an AAC utterance is a **different utterance**. The editor refuses at 16 characters instead; nothing truncates at render time.
  - A smaller font on the offending tile — one uniform size for all 12 is load-bearing. Variable line count reads fine; variable size reads as broken.

  The legitimate resolutions: shorten the `label` (the tile is a *handle* — the sentence lives in `says`), add a hand-set `\n`, adjust the shared tile type role, or accept the 6-tile layout. "Never reflow" absolutism resolves to "never work at 200%", which is worse.
- **Dropping the bold axis to halve the matrix.** It removes exactly the half most likely to be red, and costs one bool in the tuple.
- **`pumpAndSettle`.** Zero animation is a design rule, so it waits out animations that do not exist, carries a 10-minute default timeout, and truncates its stack trace when it fires. `pump()` for state changes, `pump(duration)` for anything timer-driven.
- **Reaching for a real database.** `flutter test` runs in a plain Dart VM where `sqlite3_flutter_libs` does nothing, and a drift stream under `FakeAsync` emits an `AsyncValue.loading` frame that a single `pump()` renders as a spinner — a green test asserting on an empty screen. Override `gridProvider` with a fixed `BoardGrid`.
